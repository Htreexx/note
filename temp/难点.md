# 项目一

难点：

```txt
在这个项目里，我觉得一个比较有代表性的难点是大报文网络协议设计。因为系统里不仅要传普通业务消息，还要传草图解析数据以及大模型生成的作战方案、决心图，这些数据通常都是体积比较大的 JSON。实际开发时我先分析到一个核心问题，就是 TCP 本身是字节流协议，没有消息边界，所以直接传大 JSON 很容易出现半包、粘包，或者一次发送不完整的情况；如果再把解析逻辑直接放到网络线程里，还会影响整体通信效率和客户端响应。

针对这个问题，我主要做了三件事。第一是优化协议设计，自定义消息头，在里面加入消息类型、包体长度、请求 ID 等字段，用长度字段来判断一条消息是否接收完整。第二是做大报文分片发送和接收组包，接收端为每个连接维护缓冲区，按包头长度循环拆包，这样能稳定处理半包和粘包问题。第三是将 I/O 线程和业务线程解耦，epoll 线程只负责收发和组包，JSON 解析和后续业务处理交给线程池去做，避免大报文阻塞网络线程。最终这套方案保证了大模型返回结果和推演数据能够稳定传输，客户端展示也更流畅。
```

#### 既然你已经在头部定义了数据长度，为什么还要加尾部分割符？如果大模型返回的 JSON 业务数据（也就是包体）中，恰好包含了你定义的‘尾部分割符’，你的解析逻辑会不会提前截断导致数据损坏？

这种设计主要是出于**双重校验**和**快速排错**的考虑。因为我遇到了一个bug，我在多个线程中同时往一个socket写数据，出现了数据混乱的情况——应用层数据包出现交叉的情况。通过数据长度是不好判断数据混乱的。所以我在数据的结尾添加了尾部分隔符。当读取完 `BodyLength` 长度的数据后，我再往后读两个字节。如果这两个字节是设定的尾部分割符，说明整个报文完整；如果不是，说明数据发生了错位。

只是把尾部分隔符作为一个校验位，并不会根据尾部分隔符来分割数据，而是通过数据长度。



#### 为什么要分片？分片的大小是如何设置的?

分片原因：

- TCP缓冲区大小的限制，无法一次存储大量的数据，所以还是无法避免多次发送

  ```C++
  std::queue<std::string> q;
      q.push("非常大的字符串数据...");
  
      // 核心操作：
      // 1. q.front() 返回队首元素的左值引用
      // 2. std::move 将其转换为右值引用
      // 3. 触发 std::string 的移动构造函数（或移动赋值运算符），将底层指针直接交接给 data
      std::string data = std::move(q.front()); 
  
      // 4. 此时队首的字符串已经被“掏空”（变成合法但未指定的状态），将其安全弹出并销毁
      q.pop();
  
      std::cout << "拿到的数据: " << data << std::endl;
  ```

  

- 并发的需求：如果一个请求的数据占满了TCP缓冲区，那么其它请求的数据就需要等待发送

分片大小:

- 如果把分包太小，我们需要频繁系统调用，这就会导致程序频繁地从用户态和内存态切换，效率很低。另一方面操作系统的内存管理和文件系统通常是以页（Page，一般是 4KB为单位）的单位的。所以数据的大小最好是 4KB 的整数倍。所以在本项目设置包大小16k

#### 优化：

- 往队列中拿数据，之前直接用的是拷贝构造，优化成移动构造(std::move)
- 发送大对象时，分片之后，发送完分片后，要即使释放该分片。做法是：将大对象切片，然后将切片存储在链表中，这样可以发送一完分片，就可以释放该分片所占的内存。之前的做法是将存储在std::vector中大对象，按切片大小发送，只有当全部切片发送完了, 才可以释放该内存
