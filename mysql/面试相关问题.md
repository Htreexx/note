### Q1 为什么mysql底层用B+树

1. 为什么不用hash表

   使用hash表我们也可以实现B+树类似的实现：将数据页用双向链表按主键的顺序组织起来，然后将页的指针和页中主键的范围作为value, 将主键Id作为key(注意这样实现的好处是避免随机I/O)。但是采用hash实现范围查询的效率比较低。例如我想查询Id>4的数据，那么hash查找需要按5、6、7、8...这样依次判断这个key是否存在。如果下一个存在是数据Id=10000的话，那么hash查找需要进行判断很多次，这样效率比较低。另一方面，hash扩容的情况，由于数据都是存储在磁盘中，在磁盘中进行扩容这样效率会很低。

2. 为什么不用二叉排序树/红黑树

   由于每一个结点都代表了一个页， 所以每访问一个结点，就会有一次磁盘I/O。而二叉排序树/红黑树，当数据量很大的时候，数高度就会很高，这样磁盘I/O的次数就会增多，效率就会降低。

3. 为什么不用B树

   B 树的每一个节点里，既存着索引键（Key），又存着真实的数据（Data，比如一整行记录）。这就导致一页 16KB 的空间，存不了几个数据就满了。为了存下百万条数据，数据页的数量会变多， 不利与顺序I/O。

### Q2 MySql和SQLite的区别

| **维度**       | **MySQL**                                                    | **SQLite**                                                   |
| -------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **运行形态**   | 独立的后台守护进程，需网络通信                               | 编译进应用程序的静态/动态库，直接读写本地文件                |
| **并发能力**   | **极高。** 支持细粒度的行级锁和 MVCC，千万用户可同时读写     | **极低。** 写操作通常会锁定整个数据库文件（即使启用了 WAL 模式，并发写能力依然极其有限） |
| **配置与运维** | **极其复杂。** 需要安装、配置参数（内存池大小、日志策略等）、管理用户权限、定期备份 | **零配置。** 无需安装，无需启动服务，无管理员账号，连上文件就能用 |
| **功能丰富度** | 极其丰富。支持存储过程、触发器、复杂的权限系统、多种存储引擎 | 相对精简。支持大部分标准 SQL，但省略了权限管理、复杂的内置函数等 |
| **存储规模**   | 支持 TB 级以上的海量数据分布存储                             | 通常用于 GB 级别的数据，文件过大会导致性能瓶颈               |
| **网络要求**   | 必须通过网络协议访问（哪怕是 localhost）                     | 完全不需要网络，纯本地磁盘 I/O                               |

### Q3 Innodb引擎和MyISAM引擎有什么区别

| **维度**                  | **InnoDB (现代主力)**      | **MyISAM (历史元老)**           |
| ------------------------- | -------------------------- | ------------------------------- |
| **事务 (Transaction)**    | **支持** (ACID)            | 不支持                          |
| **崩溃恢复 (Crash Safe)** | **支持** (依赖 Redo Log)   | 不支持 (容易损坏)               |
| **锁的粒度**              | **行锁** (默认)，表锁      | 只有**表锁**                    |
| **并发性能**              | **极高** (依靠行锁和 MVCC) | 较低 (写操作会阻塞全表)         |
| **索引结构**              | **聚簇索引** (数据即索引)  | 非聚簇索引 (索引和数据文件分离) |
| **外键约束**              | **支持**                   | 不支持                          |
| **`COUNT(\*)` 性能**      | 较慢 (全表扫描计算)        | **极快** (直接读取内置计数器)   |

### Q4 如果我想删除重复的数据，并保留其中的一行，sql语句该怎么写？

自连接删除

```C++
DELETE t1 
FROM users t1
JOIN users t2 ON t1.email = t2.email
WHERE t1.id > t2.id;
```

把 `users` 表当成两张表（`t1` 和 `t2`）来做连接（JOIN）。连接条件是 `t1.email = t2.email`，也就是把邮箱相同的数据匹配到一起。核心在 `WHERE t1.id > t2.id`：如果发现两行邮箱一样，并且 `t1` 的 `id` 比 `t2` 的大，就把大号的 `t1` 删掉。结果就是：经过两两相互厮杀，只有 `id` 最小的那个“天选之子”由于找不到比它更小的 `id`，得以存活。
