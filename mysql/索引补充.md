# 回表的代价

### 代价一：双倍的 B+ 树遍历开销（逻辑上的代价）

这个很好理解。本来查一次 B+ 树，经历 3 次磁盘 I/O 就能找到数据。

发生了回表，意味着你：

1. 先在二级索引树里查一遍（经历 3 次 I/O 找到主键 ID）。
2. 拿着主键 ID，再去主键（聚簇）索引树里**从根节点往下再查一遍**（再经历 3 次 I/O 找到完整数据）。

**时间消耗直接翻倍**，CPU 的遍历计算开销也随之翻倍。但相比于第二个代价，这还算轻的。

------

### 代价二：致命的“随机 I/O”（物理上的绝对代价，💥 面试核心考点）

这是回表最核心的痛点！为了让你彻底懂，我们想象一个场景：

假设你有一张表，主键是 `id`，你在 `age` 字段上建了二级索引。

你执行了一条范围查询：`SELECT * FROM users WHERE age > 20;`

**底层发生了什么极其可怕的事情？**

1. **顺序提取 ID（很快）：** MySQL 在 `age` 的二级索引树里，找到了 `age > 20` 的起始叶子节点，然后顺着链表一路往右读。这就叫**顺序 I/O**，磁盘磁头不需要乱跑，读起来极快。
2. **拿到的 ID 是散乱的：** 虽然 `age` 是有序的，但满足 `age > 20` 的对应主键 `id` 可能是极其散乱的！比如你拿到的 ID 列表是：`[8000, 2, 550, 9999, 15]`。
3. **拿着散乱的 ID 去回表（极慢）：** 现在，MySQL 要拿着这串散乱的 ID，去主键树里捞完整数据。
   - 找 `id=8000`，磁头跳到磁盘的区域 A，加载一个数据页。
   - 找 `id=2`，磁头赶紧狂奔到磁盘的区域 B，加载另一个数据页。
   - 找 `id=550`，磁头又折返回区域 C……

这就是让数据库闻风丧胆的 **随机 I/O**！在传统的机械硬盘上，磁头频繁寻道的耗时是极其恐怖的（哪怕在固态硬盘 SSD 上，随机读也比顺序读慢得多）。

------

### 💡 面试高光时刻：由“代价”引出的两个高级现象

因为回表的“随机 I/O”代价太大，MySQL 底层的**查询优化器 (Optimizer)** 非常聪明，它会根据这个代价做出两个让你意想不到的决定，这也是面试极其爱考的“坑”：

#### 1. 为什么我的索引明明建了，却“失效”了？

如果你执行 `SELECT * FROM users WHERE age > 0;`（假设表里 90% 的人年龄都大于 0）。

优化器在执行前会算一笔账：

- **方案 A（走索引）：** 去二级索引拿到 90% 的散乱 ID，然后回表引发疯狂的**随机 I/O**。
- **方案 B（全表扫描）：** 根本不理会你建的 `age` 索引，直接去主键树的最底层，顺着链表从头到尾扫一遍。虽然读的数据多，但是是极快的**顺序 I/O**。

**结果：优化器会果断抛弃索引，选择全表扫描！**

这就是面试常问的：“范围查询什么时候会索引失效？” 答案就是：**当查询出来的数据量占全表比例过大（通常超过 20%~30%），回表的随机 I/O 成本超过了全表扫描的顺序 I/O 成本时，索引就会失效。**

#### 2. MySQL 是怎么缓解回表代价的？（MRR 优化）

*(抛出这个词，面试官会觉得你学得很深)*

既然拿着散乱的 ID 回表会造成随机 I/O，MySQL 5.6 引入了一个极牛的优化技术叫 **MRR (Multi-Range Read)**。

它的思路很简单：MySQL 在二级索引拿到那串散乱的主键 ID `[8000, 2, 550, 9999, 15]` 后，**先不急着去回表**。而是把这批 ID 放在内存里**排个序**，变成 `[2, 15, 550, 8000, 9999]`。

然后再拿着排好序的 ID 去回表。这样一来，磁头在磁盘上就是顺着一个方向滑动的，成功把一部分**随机 I/O 转化为了顺序 I/O**，极大地降低了回表的代价。

------

**总结：** 回表的代价表面上是多查一棵树，**本质上是将二级索引带来的顺序 I/O 恶化成了访问聚簇索引时的随机 I/O。** 解决它的终极武器是咱们之前说的**“覆盖索引”**。

# 如果使用非主键列，而这个列也没有建立二级索引，那么用该列查找是走索引吗

**绝对不会走索引。** 这种情况会触发数据库性能的头号杀手：**全表扫描 (Full Table Scan)**。

我们可以用刚刚学过的底层原理解释一下为什么：

1. **没有二级树可查：** 既然你没有为这个列建立二级索引，那就意味着 InnoDB 底层**根本就没有为这个列去构建那棵专门的 B+ 树**。MySQL 找不到可以用来快速二分查找的“目录”。
2. **只能求助聚簇索引：** MySQL 手里现在只有唯一的一棵 B+ 树，也就是主键所在的**聚簇索引**。但是聚簇索引是按主键大小排序的，面对你这个非主键列的查询条件，它完全帮不上忙。
3. **被迫遍历整张表：** 无奈之下，MySQL 只能降落在聚簇索引的最底层（叶子节点），从第一页的第一条记录开始，顺着双向链表和单向链表，**把表里的每一行完整数据都拉出来看一眼**，挨个比对这一列的值等不等于你要找的值，直到看完最后一条数据。

### 💡 形象的比喻

就像一本现代汉语词典，里面的字词是按拼音首字母排序的（这叫聚簇索引）。

如果我现在让你找“所有带偏旁部首『氵』的字”（非主键列查询）：

- **建了索引：** 相当于词典前面有一页“部首检字表”，你翻开目录一秒钟就能找到。
- **没建索引（你的提问场景）：** 词典前面没有部首目录。你毫无办法，只能从词典的第一页第一行，一直肉眼翻到最后一页最后一行。

### 🛡️ 面试实战防坑指南

在实际开发和面试中，这种查询是非常危险的。如果表里只有几千条数据，全表扫描瞬间就查完了；但如果表里有上千万条数据，这种查询会直接把数据库的 CPU 和 I/O 资源打满，导致系统卡死。

**如何排查这种坑？**

在面试中，如果被问到“如何发现全表扫描”，标准答案是使用 **`EXPLAIN`** 关键字。

你在 SQL 语句前面加上 `EXPLAIN`（例如 `EXPLAIN SELECT * FROM users WHERE unindexed_column = 'xxx';`）。

如果在输出的执行计划中，**`type` 这一列的值显示为 `ALL`**，这就代表发生了全表扫描，是后端开发中必须通过加索引来消灭的恶性慢查询。

# limit关键字

### 一、 基本语法与用法

`LIMIT` 后面可以跟一个或两个数字参数（参数必须是非负整数）。

#### 1. 单参数用法：提取前 N 条记录

如果你只传一个数字，它代表**返回的最多行数**。

SQL

```
-- 查询年龄最大的前 5 名用户
SELECT * FROM users ORDER BY age DESC LIMIT 5;
```

#### 2. 双参数用法：实现分页 (Offset, Count)

这是 `LIMIT` 最常用的姿势。语法为 `LIMIT 偏移量, 返回行数`。

- **偏移量 (Offset)：** 指跳过前面多少行数据（注意：第一行的偏移量是 0，不是 1）。
- **返回行数 (Count)：** 指跳过之后，往后抓取多少行数据。

SQL

```
-- 跳过前 10 行，从第 11 行开始，取 5 条数据
SELECT * FROM users LIMIT 10, 5; 
```

**💡 分页公式：**

假设你在写一个 C++ 后端接口，前端传来了“当前页码 (`page_num`)”和“每页显示条数 (`page_size`)”。计算 `LIMIT` 参数的公式永远是：

- `Offset = (page_num - 1) * page_size`
- `Count = page_size`

------

### 二、 🚀 面试进阶考点：深度分页问题 (Deep Pagination)

如果在面试中你只答出 `LIMIT` 是用来分页的，只能算及格。面试官通常会紧接着抛出一个地狱级难度的问题：

**“如果表里有上千万条数据，我要查第 100 万页的数据，也就是 `LIMIT 10000000, 10`，为什么会非常慢？该怎么优化？”**

#### 为什么慢？

MySQL 执行 `LIMIT 10000000, 10` 的底层逻辑是：去聚簇索引里把前 10,000,010 条完整的数据全都查出来，然后**丢弃掉**前面的 10,000,000 条，只把最后的 10 条返回给你。

提取一千万条完整的记录，必然产生大量的磁盘 I/O 和回表操作，性能直接崩溃。

#### 怎么优化？（面试标准答案）

**优化方案 1：利用覆盖索引 + 子查询 (或 JOIN)**

思路：既然查完整记录（回表）慢，我们就先只查主键 ID，因为主键在单棵树里，并且索引是覆盖的，速度极快。找到那 10 个目标 ID 后，再回表查完整数据。

SQL

```
-- 优化前的慢查询：
-- SELECT * FROM users ORDER BY id LIMIT 10000000, 10;

-- 优化后的快查询（利用 JOIN）：
SELECT u.* FROM users u 
INNER JOIN (
    -- 子查询内部只查 ID，触发覆盖索引，速度极快
    SELECT id FROM users ORDER BY id LIMIT 10000000, 10
) AS sub 
ON u.id = sub.id;
```

**优化方案 2：记录上一页的最大 ID (游标法 / 瀑布流)**

思路：App 里的瀑布流（像刷抖音、朋友圈那样不断往下滑）通常用这种方式。前端每次请求下一页时，把上一页最后一条记录的 ID 传给后端。

SQL

```
-- 假设上一页最后一条数据的 ID 是 10000000
-- 这样写直接利用 B+ 树的快速定位，彻底避开了大偏移量
SELECT * FROM users WHERE id > 10000000 ORDER BY id LIMIT 10;
```

掌握了 `LIMIT` 的基本用法和深度分页的优化策略，关于数据提取和分页的考点你就可以轻松拿下了。
